# 情报信息审核退回功能前端交互改造计划

## 背景

本计划针对 `src/views/ctngencargo/intelligence` 情报管理模块，结合“网格员提交 - 网格长审核 - 片区长审核 - 退回 / 撤回 / 修改后通过”的流程，规划前端交互改造范围。

说明：路由和权限已完善，本计划不再展开路由、菜单、权限配置改造，主要关注页面交互流程、状态展示、按钮显隐、弹窗行为和审核记录展示。

## 当前模块状态

当前模块已经有：

- 外层页面：`intelligence_info.vue`
- 列表入口：`intelligence_table.vue`
- 统一增删改查接口：`api/unified.js`
- CMM 旧审核接口：`api/cmm.js`
- CMM 简单审核字段：`state_flag / back_reason / back_username`
- BB 新增、编辑、查看、报送字段，但暂无审核流

当前组件结构：

```text
intelligence_info.vue
└── intelligence_table.vue
    ├── cmm/add.vue
    ├── cmm/edit.vue
    ├── cmm/view.vue
    ├── bb/add.vue
    ├── bb/edit.vue
    └── bb/view.vue
```

本次改造重点是让列表、查看页、编辑页围绕审核状态形成完整交互闭环。

## 交互目标

- 列表页可直观看到每条情报的审核状态。
- 查询区和表格之间增加一行状态 / 消息通知区。
- 有消息通知时，在通知区显示。
- 有退回原因时，退回理由红色突出显示。
- 根据审核状态控制新增、删除、修改、提交、撤回、审核按钮显示。
- 网格长审核本网格点数据。
- 片区长审核本片区数据。
- 查看页可完成审核、退回、撤回等动作。
- 编辑页可支持退回后修改再提交，以及审核人修改后通过。
- BB / CMM 两类情报保持一致审核交互，业务字段差异保留。

## 状态口径

以补充任务说明为准，前端优先识别后端 `FLAG` 状态：

- `FLAG = -1`：退回
- `FLAG = 1`：网格长审核
- `FLAG = 2`：片区长审核

需要在 `constants.js` 中维护状态文案和标签样式：

```js
export const AUDIT_FLAG = {
    RETURNED: -1,
    WAIT_GRID: 1,
    WAIT_AREA: 2,
}

export const AUDIT_FLAG_TEXT = {
    [AUDIT_FLAG.RETURNED]: 退回,
    [AUDIT_FLAG.WAIT_GRID]: 网格长审核,
    [AUDIT_FLAG.WAIT_AREA]: 片区长审核,
}
```

说明：

- 如果后端字段仍叫 `state_flag`，前端需要做兼容读取，但页面语义统一叫审核状态。
- 如果后端后续补充“已办结 / 已生效 / 草稿”等状态，再追加到 `AUDIT_FLAG`。
- 现有 `BACK_STATUS_TEXT / BACK_STATUS_OPTIONS` 需要调整到新状态口径，避免旧语义 `1=修改待审、2=审核通过、-1=已退回` 和新流程冲突。

## 列表页交互

修改 `intelligence_table.vue`。

### 查询区

- 增加审核状态筛选，选项来自 `AUDIT_FLAG_TEXT`。
- 保持现有业务类型、日期、港口、评估项等查询条件不变。
- 查询按钮触发后，审核状态随列表请求参数传给后端。

### 通知区

在查询条件和表格之间新增一行通知 / 状态展示区。

展示规则：

- 有消息通知时，显示通知内容。
- 当前筛选结果存在退回记录时，可提示存在退回数据。
- 退回理由必须红色显示。
- 无通知、无退回提示时，该区域尽量不占用明显空间。

建议抽组件：

```text
src/views/ctngencargo/intelligence/components/notice-bar.vue
```

### 表格列

- 增加“审核状态”列。
- `FLAG = -1` 显示“退回”，建议红色标签。
- `FLAG = 1` 显示“网格长审核”。
- `FLAG = 2` 显示“片区长审核”。
- 有退回原因时，在状态列或相邻列展示退回原因摘要，红色显示并支持 tooltip。

### 操作按钮

操作按钮从现在的“查看 / 编辑 / 下载 / 删除”扩展为：

- 查看
- 编辑
- 提交
- 撤回
- 审核
- 修改后通过
- 下载
- 删除

按钮显示规则：

- `FLAG = -1`：显示退回状态和红色退回原因；责任节点可编辑并重新提交。
- `FLAG = 1`：网格长可审核；网格员可在网格长审核前撤回。
- `FLAG = 2`：片区长可审核；网格长可在片区长审核前撤回。
- 非当前节点人员只显示查看、下载等只读操作。
- 删除、编辑在审核中是否允许，需要以后端接口规则为准；前端先按审核中禁用删除、禁用普通编辑处理。

## 查看页交互

修改：

- `cmm/view.vue`
- `bb/view.vue`

### 基础展示

- 展示情报基础字段和附件信息。
- 展示审核状态。
- 展示退回人、退回时间、退回原因。
- 退回原因使用红色展示。
- 展示审核记录 / 修改痕迹入口或直接展示记录列表。

### 审核动作

查看页底部根据状态显示动作按钮：

- `FLAG = 1` 且当前人为网格长：显示“通过”“退回”“修改后通过”。
- `FLAG = 2` 且当前人为片区长：显示“通过”“退回”“修改后通过”。
- `FLAG = 1` 且当前人为提交网格员：显示“撤回”。
- `FLAG = 2` 且当前人为网格长：显示“撤回”。
- 其他情况仅显示“返回”。

### 退回弹窗

退回时弹出退回原因弹窗：

- 退回原因必填。
- 支持多行输入。
- 确认前二次提示。
- 提交成功后关闭弹窗，返回列表并刷新。
- 提交失败时保留输入内容，便于重试。

### 通过确认

审核通过时弹出确认框：

- 网格长通过后，状态进入片区长审核。
- 片区长通过后，进入后端定义的完成状态。
- 成功后刷新列表。

### 修改后通过

点击“修改后通过”时：

- 进入对应编辑页。
- 编辑页带入审核模式 `gridApproveWithEdit` 或 `areaApproveWithEdit`。
- 保存成功后直接调用“修改后通过”接口。
- 需要记录修改痕迹。

## 编辑页交互

修改：

- `cmm/edit.vue`
- `bb/edit.vue`

需要区分三种编辑场景：

- 普通编辑：网格员新增或退回后修改。
- 网格长修改后通过：编辑后流转到片区长审核。
- 片区长修改后通过：编辑后完成审核或进入后端定义的完成状态。

建议新增 prop：

```js
mode: normal | gridApproveWithEdit | areaApproveWithEdit
```

按钮文案建议：

- `normal`：保存、提交审核。
- `gridApproveWithEdit`：保存并通过。
- `areaApproveWithEdit`：保存并通过。

交互规则：

- 退回原因在编辑页顶部红色展示，帮助修改人定位问题。
- 修改后通过必须走审核接口，不只是普通 `editIntelligence`。
- 附件替换流程保持现有逻辑，但保存后流转动作要和当前 `mode` 匹配。
- 保存失败时停留当前页，不清空表单和附件状态。

## 新增页交互

修改：

- `cmm/add.vue`
- `bb/add.vue`

当前按钮叫“提交”，但实际更接近新增保存。建议根据接口能力确定交互：

- 如果新增即提交审核：按钮保留“提交”，保存后 `FLAG` 进入 `1`，即网格长审核。
- 如果后端支持草稿：按钮拆成“保存草稿”和“提交审核”。

从流程图看，更像“创建 / 编辑数据”后再“网格员提交”，建议优先采用“保存草稿 + 提交审核”交互；若接口任务 `68705` 不支持草稿，则按新增即提交处理。

## 组件抽取

建议新增：

```text
src/views/ctngencargo/intelligence/components/audit-dialog.vue
src/views/ctngencargo/intelligence/components/audit-records.vue
src/views/ctngencargo/intelligence/components/notice-bar.vue
```

`audit-dialog.vue` 负责：

- 通过确认
- 退回原因填写
- 撤回确认
- 修改后通过确认

`audit-records.vue` 负责展示：

- 操作人
- 操作角色
- 操作时间
- 操作结果
- 退回原因
- 修改痕迹

`notice-bar.vue` 负责展示：

- 查询区和表格之间的一行状态 / 消息通知
- 退回理由红色展示
- 空状态时最小化占位

## 接口联调点

需要后端确认：

- 状态字段是否统一叫 `FLAG`，还是列表仍返回 `state_flag`。
- `FLAG = -1` 时，如何区分是网格长退回给网格员，还是片区长退回给网格长。
- BB 是否也返回 `back_reason / back_username` 或等价字段。
- 消息通知字段来源是什么，是否随列表返回，还是单独接口获取。
- 撤回接口是否区分网格员撤回和网格长撤回。
- 修改痕迹是后端返回 diff，还是只返回操作日志。
- “已生效 / 办结”是否有独立 FLAG 状态码。
- 删除、编辑按钮在 `FLAG = -1 / 1 / 2` 下的后端允许规则。

## 推荐开发顺序

1. 修改 `constants.js`，统一 `AUDIT_FLAG / AUDIT_FLAG_TEXT` 和状态样式。
2. 改造 `intelligence_table.vue`：审核状态筛选、通知区、审核状态列、退回原因展示。
3. 新增 `notice-bar.vue`。
4. 新增 `audit-dialog.vue`，统一通过、退回、撤回确认交互。
5. 改造 CMM 查看页，替换旧审核按钮逻辑。
6. 改造 BB 查看页，补齐审核状态和审核动作。
7. 改造 BB / CMM 编辑页，支持退回后修改和“修改后通过”。
8. 新增 `audit-records.vue`，展示审核记录和修改痕迹。
9. 按接口任务 `68705` 联调提交、撤回、通过、退回、修改后通过接口。
10. 手工走通网格员、网格长、片区长三类角色的完整流程。

## 验收标准

- [ ] 列表查询条件支持审核状态筛选。
- [ ] 查询条件和表格之间可展示一行状态 / 消息通知。
- [ ] 有退回原因时，列表和详情页均红色展示退回理由。（待定）
- [ ] `FLAG = -1` 显示为退回。
- [ ] `FLAG = 1` 显示为网格长审核。
- [ ] `FLAG = 2` 显示为片区长审核。
- [ ] 网格长审核本网格点数据。
- [ ] 片区长审核本片区数据。
- [ ] 增、删、改、提交、撤回、审核、修改后通过按钮根据当前状态正确显示。
- [ ] 退回原因必填，提交失败后不丢失输入。（可能不需要草稿/缓存，丢了就丢了）
- [ ] 修改后通过会保存数据并记录修改痕迹。
- [ ] BB / CMM 两类情报流程一致，业务字段差异保留。

## 核心原则

交互判断围绕 `FLAG` 和当前操作节点收敛处理。列表负责状态感知和动作入口，查看页负责审核确认，编辑页负责修改和修改后通过，避免把流程判断分散在多个页面里重复实现。
