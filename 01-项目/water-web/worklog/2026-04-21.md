## 详细变更说明

### 1. ArcGIS 编辑页正式接入公司节点独立图层

今天最主要的一组提交是 `05aef3a`，核心目标就是把“公司节点”从普通节点体系里单独拉出来，作为 ArcGIS 编辑页的一张独立业务图层处理。

这组改动里做了几件关键事：

- 在地图层定义中新增“公司节点图层”，并明确不并入“节点图层组”
- 查询侧直接复用“公司节点资料”已有的 `GDotComp` 接口，而不是新开地图专用接口
- 表单侧复用“公司节点资料”现成 `form.vue`，避免维护第二套字段定义
- 地图交互上补齐新增、修改、删除、拖拽更新，经纬度直接回写公司节点资料
- 浏览态点击公司节点不再弹详情，不再误导性显示手型指针

这意味着 ArcGIS 编辑页现在已经不是“能看公司节点”，而是把公司节点作为一套完整可维护业务对象接进来了。

### 2. ArcGIS 图层刷新与交互逻辑被拆得更干净

这组改动里另一个很值钱的点，不是单纯把点画出来，而是把公司节点图层和原有地图刷新逻辑彻底解耦。

之前图层切换一旦走错，就容易出现两个问题：

- 切公司节点时把线、区域、整页 loading 一起带起来
- ArcGIS 原生 Layer 对象进了 Vue `reactive` 后触发 `__accessor__` 代理冲突

这次的处理方式比较稳：

- 点图层刷新改成局部刷新，不再默认调用整页 `initData()`
- 区域图层保留自己的刷新路径，避免被点图层联动误伤
- ArcGIS Layer 对象在进入状态前统一做 `markRaw`，规避 Vue Proxy 冲突
- 公司节点进入绘图后的操作面板单独收口成三按钮布局，不再沿用普通节点那套残缺模板

这类改动表面看是交互修修补补，实际上是把地图页状态边界重新理顺了，后面再加地图业务不会那么容易互相踩。

### 3. 首页大屏把公司节点和加水口正式拆层

提交 `f2473e2` 解决的是首页地图层职责混杂的问题。

原来的首页大屏里，公司节点详情轮播和加水口静态点位共享一套点层思路，这会带来两个问题：

- 公司节点详情交互容易被加水口这类纯展示点位干扰
- 后面如果想单独控制加水口显隐、样式或筛选，耦合会越来越重

这次调整后：

- 公司节点继续作为详情面板驱动层使用
- 加水口独立成单独图层，只承担静态展示职责
- 加水口改用矢量菱形点，不再复用公司节点的红色圆点交互语义
- 首页点击和 hitTest 逻辑只对公司节点详情层生效

这样首页地图的业务语义就清楚了：公司节点负责“看详情”，加水口负责“看分布”，两边不再互相拖累。

### 4. 管线管理与水质检测页做了两处小而必要的收口

今天还有两个体量较小但方向正确的修正：

- `d5a0956`：管线管理页前端缓存查询逻辑优化
- `a98e32c`：水质检测记录页按钮顺序调整

管线管理这笔提交虽然改动不大，但意图很明确，就是把查询缓存、结果复用和页面状态的一致性再理顺一遍，减少“查的是旧条件、页面显示的是新状态”这种很烦的前端毛病。

水质检测页那笔改动则更偏交互层面：把删除操作往后放，让常用操作和高风险操作分开，属于不复杂但很有必要的页面整理。

这两笔虽然不是今天的大头，但都属于“把页面从能用往顺手和稳一点再推一步”。

## 遇到的问题

### 问题1：ArcGIS 图层对象进入 Vue 状态后触发代理冲突

- **现象**：切换公司节点图层时，控制台出现 ArcGIS `__accessor__` 相关报错，导致 watcher 执行异常
- **解决方案**：将 ArcGIS 原生 `Layer` 对象在写入状态前统一做 `markRaw` 处理，避免被 Vue Proxy 代理

### 问题2：公司节点图层切换会误触发整页刷新与 loading

- **现象**：打开公司节点图层时会连带重刷点、线、区域，页面出现不必要的 loading 和闪动
- **解决方案**：把刷新逻辑拆成点图层局部刷新、区域局部刷新和全量初始化三条路径，切换公司节点时只刷新当前点层

### 问题3：首页大屏中公司节点与加水口职责耦合

- **现象**：公司节点详情交互和加水口静态展示共享点层思路，后续显隐和交互都容易互相污染
- **解决方案**：首页把公司节点与加水口拆成两张独立图层，并为加水口定义单独的矢量点样式

### 问题4：公司节点绘图面板删除无关按钮后留下空白布局

- **现象**：进入公司节点图层绘图模式后，普通节点面板删掉部分按钮，但分隔线和换行还保留，导致界面断裂
- **解决方案**：为公司节点单独渲染“新节点 / 修改 / 删除”三按钮布局，不再复用普通节点模板片段

## 技术决策

- **决策**：ArcGIS 编辑页直接复用“公司节点资料”模块已有接口与表单，不新增地图专用 CRUD 表单
- **原因**：避免维护两套字段模型，减少后续接口变更和表单校验分叉

- **决策**：公司节点图层独立于“节点图层组”存在
- **原因**：公司节点的业务语义、交互方式和普通节点不同，继续挂在节点图层组里只会增加耦合

- **决策**：公司节点浏览态不做详情弹窗和点击反馈
- **原因**：该图层当前需求只需要展示和编辑，不需要详情阅读，保留点击反馈反而会误导用户

- **决策**：首页将加水口定义为单独静态图层
- **原因**：公司节点详情交互和加水口分布展示是两类职责，拆层后显隐、样式和 hitTest 边界都更清晰

- **决策**：水质检测页先通过按钮顺序优化降低误触，而不额外引入复杂确认流
- **原因**：问题本质是操作优先级排序不合理，先用最小改动修正即可
